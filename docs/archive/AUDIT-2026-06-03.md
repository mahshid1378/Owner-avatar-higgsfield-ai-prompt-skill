## 1. Security

Threat model: local tools run by the owner (or Claude on the owner's behalf). No
multi-user/network input. No `eval`/`exec`/`pickle`/`yaml.load`/`shell=True`, no
hardcoded secrets, JSON loaded safely, subprocess uses argv-list (no shell).
**Nothing exploitable found.**

| Sev | Finding | Location |
|---|---|---|
| Low | Release slash-command interpolates `$ARGUMENTS` straight into `git commit` / `gh release create` shell snippets with no validation. Near-zero risk (owner types the version), but a malformed string with `;`/backticks could inject. **Fix:** validate `^[0-9]+\.[0-9]+\.[0-9]+$` first. | `.claude/commands/release.md:13-16` |
| Low | `seedance_lint.py` mutates `sys.path` then imports `higgsfield_memory`; only adds the script's own dir, so safe unless the repo is already writable (game over anyway). | `seedance_lint.py:399-404` |

`.claude/settings.json` grants broad permissions (`Bash(python3 *)`, `git push *`,
`gh *`, `Edit(*.py/*.md)`) — appropriate for a trusted solo repo, worth being aware of.

## 2. Bugs (Python tooling)

The module `higgsfield_memory.py` has a clean JSON error contract, but several paths
bypass it with bare `dict[key]` subscripts that crash with raw tracebacks on
malformed-but-valid JSON.

| Sev | Bug | Location |
|---|---|---|
| High | `next_id` does `int(id.split("-")[-1])` — **ValueError crash (reproduced)** on any id with a non-numeric suffix; uncaught, breaks the error contract. | `higgsfield_memory.py:63` |
| High | Frontmatter `version:`/`updated:` regex matches the **first occurrence anywhere** in the frontmatter, not anchored under `metadata:`. Works today only by luck of no collision; fragile single-source-of-truth shared by validate + PDF gen. **Fix:** anchor to `metadata:` or `yaml.safe_load`. | `validate.py:132`, `generate_user_guide.py:83` |
| Medium | `check_json_db` assumes every entry is a dict → `AttributeError` on malformed DB. Guard with `isinstance`. | `validate.py:107-111` |
| Medium | `update_filter`/`update_quality` use bare `entry["id"]` → `KeyError` on entry missing id. | `higgsfield_memory.py:188,204` |
| Medium | Bad `updated:` date escapes to exit-1 catch-all instead of documented exit-2 `FrontmatterError`. | `generate_user_guide.py:274` |
| Medium | `check_relative_paths` regex misses bare `` `vocab.md` `` (no slash) root refs — a broken link to a root file passes validation. | `validate.py:79` |
| Low | PDF output uses **relative path** `docs/user-guide/USER-GUIDE.pdf` — only works with CWD=repo root. `REPO_ROOT` constant already exists at line 32. | `generate_user_guide.py:1338` |
| Low | Seedance lint FAIL rules `\bye\b`/`\bdrake\b` false-positive on archaic "ye" ("ye gods") and the bird/verb "drake" — blocks generation. Move to WARN or add context. | `seedance_lint.py:55` |

## 4. Content & dispatcher gaps

**HIGH — `higgsfield-gpt-image-2` is orphaned from the dispatcher.** The skill is
fully built (v1.1.0, 3 files, complete frontmatter) and cross-referenced by canvas,
content-factory, and marketing-studio — but appears in **zero** rows of root
`SKILL.md`'s routing table or auto-load table. A user asking "write me a GPT Image 2.0
prompt" has **no path** from the dispatcher; it's only reachable once already inside
another skill. GPT Image is even named in the root trigger description. **Fix:** add 2
rows to `SKILL.md`. *(Also: line 59 lists "GPT Image 1.5" while the built skill is
2.0 — stale model string.)*

Other findings:

- **[Medium]** README structure tree + feature bullets omit 4 sub-skills: `canvas`, `content-factory`, `gpt-image-2`, `marketing-studio`. A reader browsing the README never sees them.
- **[Low]** `validate.py`'s `FRONTMATTER_REQUIRED` only checks `name`/`description`/`user-invocable` — it does **not** enforce `metadata.{version,updated,parent}` that CLAUDE.md mandates. Present everywhere today, but unguarded.
- **[Low]** `validate.py` doesn't cross-check **CHANGELOG.md's top version** against SKILL.md (it checks SKILL ↔ README badge ↔ footer). A release that forgets the CHANGELOG header would pass.
- **[Low]** Photodump is a named root trigger but has **no routing row** in the dispatcher.
- **[Info]** Per-skill versions span 1.0.0 → 3.5.0 — **intentional** independent versioning, not drift. Worth one line in CLAUDE.md so a future maintainer doesn't "fix" them.
- **[Cosmetic]** `MODELS-DEEP-REFERENCE.md:278-279` — 2 prose refs lack `../` prefix.

All 25 other sub-skills route correctly; all `.claude/rules/*.md` pointers resolve;
no broken links in shipped content (the ~50 "broken" links are intentional historical
citations in CHANGELOG/docs and internal planning notes).

## 5. User-perspective improvements

1. **Discoverability of GPT Image 2.0** — fix the orphan (above); it's the most capable image sub-skill and currently unreachable from the front door.
2. **No human-facing capability index.** The dispatcher's two tables are agent-facing. There's no single "here's everything this skill can do" map for a person deciding what to ask. A short capability index in README would help.
3. **README is behind the codebase** — refresh the structure tree + feature list to all 26 sub-skills (Canvas, Content-Factory, GPT Image, Marketing Studio missing).
4. **Thinly-wired reference files** — `production-benchmarks.md` (2 refs) and Photodump have weak routing despite being named triggers.

## 6. Developer-perspective improvements

1. **[Medium] No CI.** There's no `.github/` at all. `validate.py`/`validate_user_guide.py` run only when a contributor remembers. A trivial `.github/workflows/validate.yml` (`pip install -r requirements.txt && python3 validate.py`) on PR would auto-catch frontmatter/path/version regressions — high value given the release discipline.
2. **[Medium] No orphan-skill guard.** A check that every `skills/higgsfield-*/` dir appears in root SKILL.md (and vice-versa) would have caught the gpt-image-2 gap. **Single highest-leverage validation addition.**
3. **[Low] Release process is partly manual** — version lives in 4 hand-edited spots (SKILL frontmatter, README badge, README footer, CHANGELOG header). A small bump script setting all four from one arg would remove drift risk; validate.py only catches it after the fact.
4. **[Low] Testing is manual-only by design** (CONTRIBUTING.md) — reasonable for a prompt library, but no routing snapshot/fixture exists.

## 7. Docs hygiene & consolidation plan

**KEEP (live, current):**

- `docs/Seedance 2 Skill.md` — **not** redundant with the seedance sub-skill. It's a distinct bilingual EN+ZH JSON director API; load-bearing, referenced by 3 skills + vocab.md.
- `docs/user-guide/USER-GUIDE.pdf` + **exactly one** baseline (`...baseline-v3.8.1`, the only file the validator actually diffs against).

**CONSOLIDATE (4 stale, paraphrase-only audit/inventory docs — zero live references):**

- `docs/archive/v3.0.0/v3-audit-report.md` (v3.0.0)
- `docs/archive/v3.0.0/v3-inventory.md` (v3.0.0)
- `docs/pdf-audit/AUDIT-REPORT.md` (v3.4.0)
- `docs/pdf-audit/AUDIT-REPORT-v3.6.0.md` (v3.6.0)

Fold into a single `docs/archive/HISTORY.md` with dated chronological sections and a
preamble ("point-in-time historical records; current state lives in root `.md` +
CHANGELOG"). Then delete the empty `v3.0.0/` and `pdf-audit/` dirs and update the
README doc tree.

**One blocker first:** `validate_user_guide.py:56` has a live docstring pointer to
`AUDIT-REPORT-v3.6.0.md` (drift items D3–D8). Inline that one-sentence summary into
the docstring before moving the file, so the validator stops depending on an archived
audit.

**PRUNE (binary bloat):**

- 13 of 14 `USER-GUIDE.pdf.baseline-*` PDFs (~730K) are **read by no tooling** — only `DEFAULT_BASELINE` (the latest) is consumed. The "baselines accumulate" convention is purely manual; every old baseline is recoverable via `git show <tag>:docs/user-guide/USER-GUIDE.pdf`. **Recommend:** `git rm` the 13 superseded ones, switch convention from *accumulate* → *rotate* (replace baseline each release), and update the CHANGELOG/MEMORY note that codified accumulation.
- `.DS_Store` files in `docs/` are already gitignored/untracked — harmless, no git action needed.
